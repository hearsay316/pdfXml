# 2026-04-06 工作日志

## FreeText 中文显示修复（第二轮 — 根本解决）

**问题**：FreeText 注解中的中文内容在 PDF 中不显示。

### 根因分析（3 个 bug）
1. **DA 字符串被截断**：`replace_font_in_da()` 函数替换 `/Helvetica` 为 `/STSong-Light` 时，把字体大小 `16 Tf` 也吃掉了，导致 DA 变成 `/STSong-Light )` 而非正确的 `/STSong-Light 16 Tf`
2. **缺少 AP (Appearance Stream)**：所有 FreeText 注解都没有外观流。PDF 阅读器依赖 AP 流来渲染注解内容，没有 AP 就什么都不显示
3. **DR 字典存在但 AP 不存在**：即使 DR 字体资源正确注册了，没有 AP 流也无法渲染

### 修复内容
1. **重写 `replace_font_in_da()` 函数**：使用正则表达式精确匹配 `/FontName size Tf` 模式，保留原始字号
2. **新增 `build_freetext_ap_stream()` 方法**：为每个中文 FreeText 注解生成 Form XObject 类型的 AP 外观流
   - 包含完整的 PDF 图形指令序列：`q` → 颜色设置 → 字体设置 → `BT` → 文本显示(`Tj`) → `ET` → `Q`
   - 中文文本以 UTF-16BE 十六进制编码写入流中
3. **AP 流结构**：`/AP << /N << /Type/XObject /Subtype /Form /BBox[0 0 100 20] >> >>`

### 第三轮修复 — AP Stream 内容 bug（Tf 位置 + 颜色分量）
4. **Tf 在 BT 外部**：`build_freetext_ap_stream()` 把字体设置写在 `BT` 前面且缺少 `Tf` 关键字 → 移到 BT 内部
5. **颜色只有 2 个分量**：RGB 解析只取 `rg` 前 2 个数字，应为 3 个 (r g b) → 修复为取 3 个

### 使用方式修正
**关键发现**：程序有 2 种模式！
- **新建模式**（之前用的）：`--input xml -o out.pdf` → 创建空白 PDF + 注解 → 只有 67KB
- **合并模式**（正确的）：`--input xml --target-pdf original.pdf -o out.pdf` → 基于 15MB 原始 PDF 添加注解
- 原始 PDF `WTchbOFP.pdf` 有 71 页、15.3MB、380 个流对象

### 最终验证结果
- 文件大小：**15.37 MB**（基于原始 PDF 合并）
- 页数：**71 页**（含原始文档内容）
- FreeText：14 个，**全部有 AP 流**
- DA 格式正确：`/STSong-Light 16 Tf`
- AP 流结构正确：`q → rgb → BT → /Font size Tf → Tj → ET → Q`
